# 2. Účel playbooku
Tento playbook definuje standardizovaný postup pro detekci, triage, containment a eradikaci incidentů spojených s kompromitací uživatelského nebo servisního účtu (Compromised Account). Pokrývá scénáře, kdy útočník získal platné přihlašovací údaje a aktivně je zneužívá v prostředí zákazníka — bez ohledu na to, jak ke kompromitaci došlo (phishing, brute force, credential leak, malware, insider threat).
Klíčová hodnota playbooku: Kompromitovaný účet je nejčastější vstupní bod pro ransomware, APT a data exfiltration. Rychlá a správně provedená response minimalizuje lateral movement, persistence a business dopad. Playbook zajišťuje konzistentní, auditovatelný a měřitelný response proces od detekce přes revokaci přístupu po uzavření incidentu.

Service accounts a aplikační identita — API keys, managed identities, OAuth tokens
Privilegované účty — Domain Admin, Global Admin, Enterprise Admin, CISO, C-level
Účty s přístupem k citlivým systémům — finance, HR, legal, OT/SCADA integrace
Cloud workload identita — Azure Service Principal, AWS IAM role, GCP service account
### Mimo scope:
Fyzické přístupy a přístupové karty bez IT integrace
Kompromitace síťových zařízení (dedikovaný playbook NW Incident)
Čistý brute force bez potvrzené kompromitace — viz PB-04
Interní pentest nebo Red Team — ověřit před triage v Change Management
# 4. Vstupní podmínky
SIEM korelační alert: úspěšný login z anomální lokace, zařízení nebo času po předchozí failed auth sérii

MDE / Cortex XDR / Bitdefender: credential access alert — LSASS dump, Mimikatz behavior, Pass-the-Hash, Pass-the-Ticket

EDR alert: post-exploitation aktivita z kontextu uživatelského účtu (lateral movement, enumeration, persistence)

Interní alert: nová MFA zařízení zaregistrována bez vědomí IT, neznámé OAuth app consenty, inbox pravidla vytvořená automaticky
Phishing potvrzení (z PB-01): uživatel kliknul na phishing link a zadal credentials

# 6. Detekční logika
### Primární indikátory kompromitace (Compromise Indicators):
Risky Sign-In (High/Critical) v Entra ID: Unfamiliar sign-in properties, Atypical travel, Malware-linked IP, Anonymous IP (Tor)
Credential access alerts z EDR: LSASS memory read (T1003.001), Mimikatz behavior detection, credential dumping z SAM/NTDS
Pass-the-Hash (PtH) / Pass-the-Ticket (PtT): laterální pohyb pod ukradeným hash nebo Kerberos ticketem
Úspěšný login z neznámého zařízení (non-compliant, non-registered) na privilegovaný účet
Inbox rules vytvořeny automaticky: pravidla přesměrující poštu externě, do skryté složky, mazání bezpečnostních notifikací
Nová MFA zařízení registrována bez IT change ticketu — kritický persistence indikátor (T1098.005)
OAuth aplikace s vysokými oprávněními přidány bez souhlasu IT/Security (T1550.001)
Přihlášení z geo-lokace vzdálené > 500 km v < 2 hod od předchozího loginu (Impossible Travel — T1078)
Uživatel hlásí: přeposílání pošty, zprávy odeslané bez jeho vědomí, změna hesla / MFA bez jeho akce
Service account přihlášení v nestandartním čase nebo z nestandartního zdroje
### Kritické eskalační triggery — okamžité P1:
Kompromitovaný privilegovaný účet (Domain Admin, Global Admin, Enterprise Admin, CISO, C-level)
Post-kompromitace lateral movement potvrzeno EDR (PtH, PtT, WMI/RDP z kompromitovaného účtu)
Credential dumping potvrzeno — útočník extrahoval další credentials
Ransomware nebo destructive malware spuštěno pod kompromitovaným účtem
Kompromitace service accountu s přístupem k databázím, OT systémům nebo zálohovacím serverům
### SIEM korelace — Microsoft Sentinel (KQL):
// Compromised Account — Multi-signal korelace
let RiskySignIns = SigninLogs
| where TimeGenerated > ago(24h)
| where RiskLevelDuringSignIn in ('high', 'critical')
| where ResultType == 0 // successful
| project UserId, UserPrincipalName, IPAddress, Location, AppDisplayName, TimeGenerated;
let CredentialAlerts = SecurityAlert
| where TimeGenerated > ago(24h)
| where AlertName has_any ('LSASS','Mimikatz','Pass-the-Hash','credential dump','Pass-the-Ticket')
| extend AccountName = tostring(Entities[0].Name);
RiskySignIns
| join kind=leftouter (CredentialAlerts) on $left.UserPrincipalName == $right.AccountName
| extend Severity = iff(isnotempty(AccountName), 'P1-CredentialDump+RiskySignIn', 'P2-RiskySignIn')
| project TimeGenerated, UserPrincipalName, IPAddress, Location, AppDisplayName, AlertName, Severity
### SIEM korelace — Splunk (SPL):
index=entra_id sourcetype=azure:signinlogs risk_level_during_sign_in IN ("high","critical") result_type=0
| stats count as RiskyLogins, values(ip_address) as SrcIPs,
values(location.countryOrRegion) as Countries by user_principal_name
| join type=left user_principal_name [
search index=mde sourcetype=defender:alerts
category IN ("CredentialAccess","LateralMovement") status=resolved
| stats count as EDRAlerts by AccountUpn
| rename AccountUpn as user_principal_name
]
| eval Risk=if(EDRAlerts>0,"P1-PostExploit","P2-RiskyAccount")
| table _time user_principal_name RiskyLogins SrcIPs Countries EDRAlerts Risk
### SIEM korelace — FortiSIEM (pseudo-rule):
Rule: COMPROMISED_ACCOUNT_MULTI_SIGNAL
Conditions:
Event1: ENTRA_RISKY_SIGNIN severity=HIGH|CRITICAL, resultType=SUCCESS
Event2 (optional): MDE_CREDENTIAL_ACCESS targetUser=Event1.user within 4h
Event3 (optional): ENTRA_MFA_DEVICE_REGISTERED user=Event1.user within 24h
Match: ANY(Event1) + IF(Event2 OR Event3) → Severity P1
ANY(Event1) alone → Severity P2
Action: Trigger INCIDENT, XSOAR enrichment workflow, notify SOC L2
Severity Override: P1 if user in (Domain Admins, Global Admins, Service Accounts critical)
### False positive scénáře:
Uživatel přihlášen z hotelové/firemní VPN z jiné země — ověřit s uživatelem a HR (cestovní plán)
Shared service account přihlášený z více systémů — ověřit v CMDB jako autorizovaný pattern
Testovací prostředí / DEV environment s clonovanými production credentials — ověřit scope
MFA re-enrollment po výměně telefonu bez IT change ticketu — ověřit s uživatelem a helpdesk
Pentest / Red Team aktivita — povinně ověřit v Change Management před triage
# 7. Severity model

| P | Název | Technické podmínky | Business Impact | SLA odezva |
|---|---|---|---|---|
| P1 | Critical | Kompromitace privilegovaného účtu (DA/GA/EA/SA critical) / Lateral movement potvrzeno / Credential dumping z EDR / Ransomware nebo malware pod kompromitovaným účtem / AD/Entra ID tenant-wide kompromitace / Service account s přístupem k DB, zálohy, OT | Plná kompromitace identity a prostředí, potenciální AD Takeover, masivní business dopad, regulatorní hlášení | 15 min / 24/7 |
| P2 | High | Kompromitace běžného uživatelského účtu s potvrzenou aktivitou útočníka / Risky Sign-In High/Critical + úspěšný login / Impossible Travel + post-login aktivita / Nová MFA registrace nebo OAuth consent bez IT souhlasu / Credential leak potvrzeno v TI feed | Kompromitace uživatelského účtu, risk přístupu k citlivým datům nebo systémům, BEC risk | 30 min / 24/7 |
| P3 | Medium | Risky Sign-In Medium / Anomální login bez dalších signálů / Credential v breach databázi bez potvrzené aktivity / MFA push odmítnuta uživatelem, bez úspěšné auth / Podezřelé chování na účtu bez clear kompromitace | Potenciální riziko kompromitace, zatím bez potvrzeného přístupu útočníka | 2 hod / BH |
| P4 | Low | False positive pravděpodobný / Ojedinělá anomálie bez dalšího kontextu / Uživatel sám hlásí podezření bez technických indikátorů | Zanedbatelný — monitoring a ověření | 8 hod / BH |

# 9. Rozhodovací logika
### Decision tree — Compromised Account:
**Je zasažen privilegovaný účet (Domain Admin / Global Admin / Enterprise Admin / CISO / C-level)?**
ANO → P1 OKAMŽITĚ, IR Team, war room, management notification, zvážit AD forest reset
NE → pokračovat na další krok
**Je detekována post-login aktivita (inbox rules, OAuth consent, MFA device, AD object, GPO)?**
ANO → P1/P2, persistence mechanismus identifikován, IR Team, forenzní analýza
NE → pokračovat
**Je detekován lateral movement nebo credential dumping z EDR?**
ANO → P1, rozšířit containment na všechny zasažené systémy, IR Team
NE → pokračovat
**Je detekován Impossible Travel nebo login z Tor/anonymizing proxy?**
ANO → P2, okamžitý reset hesla a revokace session, blokace IP
NE → pokračovat
**Je kompromitace potvrzena z credential breach databáze nebo TI feed?**
ANO → P2, reset hesla, ověřit aktivitu účtu za posledních 30 dní
NE → P3, watchlist, uživatel informován, monitoring 24 hod
**Reaguje útočník na containment (nové přihlášení po resetu, nová MFA registrace)?**
ANO → P1, útočník má perzistenci, IR Team, kompletní clean-up
NE → pokračovat dle severity
**Je scope incidentu > 3 zasažené systémy nebo > 1 kompromitovaný účet?**
ANO → P1/P2 s IR eskalací, broad containment, management notification
NE → standardní P2 workflow
# 10. Reakční kroky
## SOC L1
Příjem alertu — ověřit uživatele v AD/Entra ID, typ účtu, skupiny, privilegia
Vytvořit ticket — P2 výchozí, upravit na P1 pokud privilegovaný účet nebo EDR alert
Entra ID: zkontrolovat Risky Sign-In report, Risk Level, Risk Detail, IP, geolokace
Ověřit přítomnost post-login aktivity: Entra ID Audit Log — inbox rules, MFA registrace, delegace
Základní enrichment: IP geolokace, ASN, VirusTotal — výsledky do ticketu
KRITICKÝ KROK: Pokud privilegovaný účet nebo EDR alert → IHNED eskalovat na L2 bez čekání na dokončení triage
Zdokumentovat: čas detekce, trigger, účet, IP, geolokace, typ kompromitace (pokud znám) do ticketu
## SOC L2
Převzetí ticketu, validace severity, deep dive analýza
Kompletní post-login audit: inbox pravidla, delegace, OAuth app consenty, MFA device registrace, Entra ID sign-in history 30 dní
EDR query: MDE Advanced Hunting / Cortex XDR / Bitdefender — aktivita pod kompromitovaným účtem posledních 72 hod
Okamžité rozhodnutí: reset hesla a revokace session (P1/P2 — provede zákazník, SOC monitoruje)
Blokace útočné IP na FW a Conditional Access Named Locations
IOC distribuce do Sentinel TI, XSOAR, FW, proxy — všechny identifikované IOC
First notification zákazníkovi — email s detailem a požadovanými akcemi (viz sekce 13)
Scope assessment: kolik systémů kompromitovaný účet přistupoval — AD audit, M365 audit log
IR Team aktivace pokud: privilegovaný účet, lateral movement, credential dumping, scope > 3 systémů
## IR Team (L3)
Forenzní analýza: kompletní timeline kompromitace — první přihlášení útočníka, všechny akce, laterální pohyb
Identifikace persistence mechanismů: inbox pravidla, MFA devices, OAuth apps, Scheduled Tasks, Registry Run Keys, nové AD objekty
Credential analysis: Které credentials byly stealnuty? Jaké systémy jsou v riziku? Pass-the-Hash domain-wide?
Lateral movement mapping: jaké systémy byly přístupny a kompromitovány pod tímto nebo ukradenými credentials
Root cause analysis: jak útočník získal první přístup — phishing, BF, credential leak, insider, supply chain
Eradikace: reset VŠECH kompromitovaných hesel, revokace tokenů, odstranění VŠECH persistence mechanismů
AD/Entra ID clean-up: ověřit všechny admin účty, service accounts, OAuth apps, MFA enrollments po incidentu
IR Report: kompletní timeline, scope, root cause, eradication steps, recommendations
## Zákazník / Customer Security
Reset hesla kompromitovaného účtu IHNED — nečekat na instrukce SOC pokud P1
Revokace Entra ID session tokenů: Revoke-AzureADUserAllRefreshToken nebo Entra ID portal
Zkontrolovat a ODSTRANIT: inbox pravidla (přesměrování, mazání), delegace, OAuth app consenty
MFA re-enrollment na čistém zařízení — pouze po potvrzení, že stávající MFA není kompromitováno
Ověřit Conditional Access politiky — MFA pro všechny přístupy privilegovaných účtů
Kontrola dalších potenciálně kompromitovaných účtů — stejná hesla, sdílené credentials
Informovat zasaženého uživatele — instrukce pro bezpečné chování, hlášení podezřelých aktivit
Dočasné zvýšení monitoringu pro účty ve stejné organizační jednotce nebo skupině

# 12. Eskalační pravidla
### L1 → L2:
Risky Sign-In High/Critical + potvrzený úspěšný login v Entra ID
Cílový účet je privilegovaný (Domain Admin, Global Admin, Enterprise Admin, Service Account kritický)
Post-login aktivita detekována: inbox pravidla, nová MFA zařízení, OAuth consenty
EDR alert v kontextu kompromitovaného účtu (credential access, lateral movement)
Credential potvrzeno v breach databázi nebo TI feed
Impossible Travel nebo login z Tor / anonymizing proxy
SLA: do 15 minut od alertu (P1/P2), do 30 minut (P3 s escalation trigger)
### L2 → L3 / IR Team:
Kompromitace privilegovaného účtu potvrzena nebo lateral movement detekován
Post-login persistence mechanismus identifikován (inbox rules, OAuth app, MFA device, AD object)
Credential dumping nebo Pass-the-Hash / Pass-the-Ticket potvrzeno EDR
Scope incidentu > 3 zasažené systémy nebo > 1 kompromitovaný účet
Ransomware nebo destructive malware signály v kontextu kompromitovaného účtu
Útočník reaguje na containment — persistence potvrzena, opakované přístupy
SLA: okamžitě při P1, do 30 minut při P2 s IR triggerem
### SOC → Zákazník (okamžitá akce):
Reset hesla kompromitovaného účtu — SOC nemůže provést bez zákaznického oprávnění
Revokace Entra ID session tokenů — zákazník provede v portálu nebo PowerShell
Ověření a odstranění persistence mechanismů — inbox pravidla, OAuth consenty
MFA re-enrollment — vyžaduje spolupráci uživatele
SLA: do 15 minut od P1/P2 potvrzení
### Major Incident / Management escalation:
Kompromitace privilegovaného účtu (DA, GA, CISO, C-level)
Tenant-wide kompromitace — více než 3 kompromitované účty nebo AD forest attack

# 19. Lessons Learned
### Co sledovat po každém P1/P2 incidentu:
Jak útočník získal první přístup? Phishing, credential leak, brute force, insider, supply chain, malware?
Bylo MFA aktivní na kompromitovaném účtu? Pokud ne → eskalovat na zákazníka jako CRITICAL gap
Jak dlouho byl útočník aktivní před detekcí? Dwell time — pokud > 2 hod → tuning MTTD
Byly nalezeny persistence mechanismy? → Přidat do SIEM detekčního modelu
Proběhl lateral movement? Kolik systémů bylo zasaženo? → Ověřit segmentaci sítě a PAM
Jak zákazník reagoval na first notification? Byl reset hesla proveden do SLA? Pokud ne → procesní doporučení
### Typické slabiny detekce:
Chybějící Entra ID Risky User integrace do SIEM → risky sign-in nekoreluje s EDR alertů
Příliš vysoký threshold pro anomální přihlášení → kompromitace z nové lokace nezachycena
Absence UEBA baseline → anomální chování uživatele pod radarem bez historického kontextu
Chybějící monitoring MFA device registrací → persistence přes nové MFA zařízení nezachycena
OAuth app consent monitoring nenasazen → persistence přes OAuth token nezachycena
Příliš dlouhé session tokeny bez revokace → útočník aktivní i po resetu hesla
### Doporučené akce po incidentu:
Owner: SIEM Engineer — přidat detekci nových MFA device registrací a OAuth app consentů do SIEM pravidel
Owner: Zákazník Security — vynutit phishing-resistant MFA (FIDO2) pro privilegované účty
Owner: Zákazník IT — implementovat PIM (Privileged Identity Management) — just-in-time admin přístup
Owner: SOC L3 — ověřit credential breach monitoring pokrytí — HIBP API nebo dark web feed integrace
Owner: SOAR Engineer — zkrátit session token lifetime pro privilegované účty v CA politice
# 20. Doporučení
### Technická doporučení:
Nasadit phishing-resistant MFA (FIDO2 / passkeys) pro VŠECHNY privilegované účty — priorita #1
Implementovat Entra ID PIM (Privileged Identity Management) — just-in-time admin přístup s approval workflow
Nasadit Microsoft Defender for Identity — on-prem AD monitoring, PtH/PtT detekce, lateral movement alerting
Implementovat Conditional Access: MFA + Compliant Device + Named Location pro VŠECHNY admin přístupy
Zkrátit session token lifetime — max 1 hod pro privilegované účty, 8 hod pro standard users
Nasadit Microsoft Defender for Cloud Apps — detekce impossible travel, OAuth consent abuse, inbox manipulation, mass download
Implementovat credential breach monitoring — HIBP API nebo dark web feed integrace do SOAR workflow
Zavést OAuth app pre-approval policy — blokovat neautorizované OAuth consenty v Entra ID
### Procesní doporučení:
Zavést pravidelný Privileged Account Review (Quarterly Access Review) — PIM/PAM implementace
Implementovat Account Lifecycle Management — automatická deaktivace účtů při odchodu zaměstnance do 24 hod
Provádět pravidelné credential hygiene audity — password age, MFA enrollment, shared credentials
Tabletop cvičení: scénář kompromitace DA s lateral movement a ransomware deployment — 1× ročně
Vyžadovat helpdesk ticket pro KAŽDOU změnu MFA zařízení nebo OAuth consent — auditovatelnost
### Governance doporučení:
Definovat Account Classification Policy: privilegovaný / servisní / uživatelský — různá úroveň ochrany
Zavést povinný MFA re-enrollment po jakékoli kompromitaci nebo suspicious MFA activity
Dokumentovat všechny service accounts s vlastníky a účelem — rotace credentials každých 90 dní
Regulatorní posouzení: každý P1 incident posoudit z hlediska NIS2/GDPR/ZKB notifikační povinnosti
### Automatizace (SOAR):
Automatizovat: Risky Sign-In → SOAR enrichment + ticket creation + P2 default do 5 min od alertu
Automatizovat: credential breach hit → okamžitá notifikace zákazníkovi + forced password reset workflow
Automatizovat: nová MFA device registrace mimo business hours → auto-suspend + L2 notification
Zavést: User Risk Score accumulation — SOAR tracking risk events per user pro long-term sledování
