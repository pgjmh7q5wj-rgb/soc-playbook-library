# 2. Účel playbooku
Tento playbook definuje standardizovaný, auditovatelný postup pro detekci, triage, containment, eradikaci a recovery při ransomware útoku. Pokrývá celý incident response lifecycle dle NIST SP 800-61r3 — od prvního EDR/XDR alertu nebo uživatelského hlášení po uzavření incidentu a lessons learned.
Ransomware je kritická hrozba s potenciálem způsobit úplné přerušení provozu, ztrátu dat, regulatorní sankce (NIS2/GDPR) a reputační škodu. Playbook zajišťuje:
Minimalizaci blast radius — rychlý containment před šířením do dalších segmentů
Ochranu záložních dat — okamžitá izolace backup systémů
Koordinovaný response — jasné role SOC, IR týmu, zákazníka a managementu
Auditovatelnost — kompletní evidence pro regulátora, pojišťovnu a forenzní šetření
Řízená komunikace — interní i zákaznická, s respektem k NIS2 notifikačním povinnostem
Playbook je určen pro SOC analytiky L1/L2/L3, IR tým, zákazníka a management. Je operacionalizovatelný v Cortex XSOAR a implementovatelný v Microsoft Sentinel, Splunk a FortiSIEM.
# 3. Rozsah použití
## V scope:
Endpoint s nainstalovaným EDR agentem: Microsoft Defender for Endpoint (MDE), Microsoft Defender XDR, Bitdefender GravityZone

Trigger: EDR/XDR alert na šifrování souborů, masové přejmenování přípon, ransom note, VSS/shadow copy deletion, service stop

Trigger: Uživatelské hlášení — soubory nepřístupné, neznámé přípony, ransom note v adresáři
Trigger: Threat intelligence hit — known ransomware group IOC (hash, IP, domain, C2)
Trigger: Backup systém alert — neočekávaná změna nebo smazání zálohy

OT/ICS prostředí — vyžaduje specializovaný playbook s OT safety protokoly

# 4. Vstupní podmínky
EDR/XDR alert: detekce ransomware rodiny (MDE: Ransom:Win32/*, Bitdefender GZone ransomware behavioral block)
EDR behavioral alert: masová změna přípon souborů (>100 souborů / 60 sekund), zápis ransom note (README.txt, DECRYPT.txt, !HELP.txt)
EDR alert: smazání Volume Shadow Copies (vssadmin delete shadows, wbadmin delete, bcdedit /set recoveryenabled No)
SIEM korelace: anomální SMB lateral movement + PowerShell execution + mass file I/O na více hostech současně
SIEM korelace: deaktivace Windows Backup, zastavení databázových a AV služeb (net stop, sc stop)
Uživatelské hlášení: soubory mají neznámou příponu, nelze je otevřít, v adresáři se zobrazuje ransom note
Threat intelligence: known ransomware C2 infrastruktura nebo affiliate TTP detekovaná v síti
Backup systém alert: neočekávané smazání nebo zašifrování zálohy, změna retence
SOAR automatický trigger z korelačního workflow (priority score ≥ 9/10 → auto P1)

# 6. Detekční logika
## 6.1  Primární indikátory — EDR/XDR behavioral detection
Masová změna přípon souborů (>100 unikátních přípon / 60 sec) — Bitdefender HyperDetect + MDE behavioral rule
Zápis ransom note do adresářů: README_[ransom].txt, DECRYPT.txt, !HOW_TO_DECRYPT.txt, recover-files.txt
Smazání Volume Shadow Copies: vssadmin.exe delete shadows /all /quiet, wbadmin delete catalog, bcdedit /set {default} recoveryenabled No
Zastavení zálohovacích a DB služeb: net stop / sc stop (MSSQL, MySQL, Veeam, BackupExec, Windows Backup)
Šifrování systémových svazků nebo síťových sdílených složek (SMB) z jednoho hostu
Anomální PowerShell / WMI execution s base64 encodingem v kombinaci s file I/O spike
Deaktivace Windows Defender / AV produktu přes GPO nebo registry key
## 6.2  SIEM korelace — pseudo-query (Sentinel KQL)

| // Sentinel KQL — Ransomware multi-signal korelace SecurityEvent | where EventID in (4688, 4104) | where CommandLine has_any ("vssadmin delete", "wbadmin delete", "bcdedit /set", "net stop", "sc stop") | join kind=inner (     DeviceFileEvents | where ActionType == "FileRenamed"     | summarize FileCount=count(), UniqueExts=dcount(FileName) by DeviceId, bin(Timestamp, 1m)     | where FileCount > 100 or UniqueExts > 20 ) on $left.Computer == $right.DeviceId | extend RiskScore = case(UniqueExts > 50, 10, UniqueExts > 20, 8, FileCount > 200, 9, true(), 5) | where RiskScore >= 8 | project Timestamp, Computer, CommandLine, FileCount, UniqueExts, RiskScore |
|---|

## 6.3  FortiSIEM / Splunk korelace
FortiSIEM rule: [Ransomware-FileEncryptionSpike] — file rename rate > threshold + shadow copy deletion na stejném hostu do 5 min
Splunk SPL: index=wineventlog EventCode=4688 | search CommandLine="*vssadmin*delete*" | join DeviceName [search index=edr ActionType="FileRenamed" | stats count as FileRenames by DeviceName | where FileRenames > 100]
## 6.4  False positive scénáře
Hromadné přejmenování souborů při legální archivaci/konverzi — ověřit s IT change management, whitelist process hash
Backup software provádějící VSS snapshot deletion po úspěšné záloze — ověřit backup job log a časový soulad
Pentest / red team cvičení se simulací ransomware — ověřit s zákazníkem před alertem, whitelist test hash
Storage migration nástroje s hromadným rename — ověřit asset owner a change ticket
# 7. Severity model

| P | Název | Technické podmínky | Business Impact | SLA odezva |
|---|---|---|---|---|
| P1 | Critical | Aktivní ransomware šifrování ≥1 hostu / ransom note nalezena / VSS smazány / lateral movement do ≥2 hostů / DC nebo kritický server zasažen / data exfiltrace (double extortion) | Masivní dopad: ztráta dat, výpadek provozu, regulatorní notifikace, potenciální reputační škoda | 15 min / 24/7 |
| P2 | High | Ransomware indikátory izolovány na 1 hostu, šifrování zastaveno EDR, C2 komunikace detekována ale nepotvrzena, podezřelý precursor activity (discovery + lateral prep) | Vysoký risk šíření, potenciální ztráta dat na zasaženém hostu, nutný okamžitý containment | 30 min / 24/7 |
| P3 | Medium | Ransomware dropper / loader detekován a v karanténě před execution, known ransomware hash detekován bez behaviorálního triggeru, podezřelý phishing leading k ransomware payload | Riziko eskalace pokud payload spuštěn, zatím žádný business dopad | 2 hod / business hours |
| P4 | Low | False positive pravděpodobný, testovací ransomware sample v sandboxu, adware s falešným ransom hlášením (scareware) | Zanedbatelný dopad | 8 hod / business hours |

**⚠  RANSOMWARE JE VŽDY VÝCHOZÍ P1 DO VYLOUČENÍ OPAKU. Downgrade na P2/P3 pouze po potvrzení, že šifrování neproběhlo a EDR karanténa zabránila exekuci.**

# 9. Rozhodovací logika
## Decision tree — Ransomware Response
**Krok 1: Je detekována masová změna přípon souborů nebo ransom note?**
ANO → P1 výchozí, pokračovat krok 2
NE → Dropper/loader pouze detekován? → P3 standardní malware workflow (PB-02)
**Krok 2: Stále probíhá šifrování (live)?**
ANO → OKAMŽITÁ izolace endpointu přes EDR API → krok 3
NE (EDR zastavil) → Endpoint izolovat, nižší urgence eradikace → krok 3
**Krok 3: Byly smazány Volume Shadow Copies nebo zálohy?**
ANO → Recovery z offline backup nebo DR plán; forenzní sběr před eradikací povinný; P1 zachovat
NE → VSS dostupné → recovery po eradikaci možný přes VSS; pokračovat krok 4
**Krok 4: Je detekován lateral movement (SMB, WMI, RDP) na další hosty?**
ANO → Izolace síťového segmentu (nejen endpointu); IR team aktivovat; war room otevřít; management alert
NE → Izolace endpointu postačuje; pokračovat krok 5
**Krok 5: Byl kompromitován privilegovaný účet (domain admin, service account)?**
ANO → Reset všech admin hesel, Krbtgt reset (2x), Golden Ticket risk; IR team forenzní analýza AD
NE → Reset hesla zasaženého uživatele; pokračovat krok 6
**Krok 6: Je detekována C2 komunikace nebo data exfiltrace (double extortion)?**
ANO → Blokovat C2 na FW/proxy; informovat zákazníka o potenciálním data leak; Legal/Compliance zapojit; GDPR/NIS2 posoudit
NE → Standardní containment; IOC distribuce; pokračovat eradikaci
**Krok 7: Existuje funkční záloha pro recovery?**
ANO → Recovery plán aktivovat po eradikaci; ověřit integritu zálohy před obnovením
NE → Zvážit decryptor dostupnost (No More Ransom); posoudit platbu výkupného — VŽDY eskalovat na management + Legal (nešéfuje SOC)
# 10. Reakční kroky
## SOC L1
Příjem alertu, ověření endpointu v CMDB, otevření P1 ticketu v Jira/ServiceNow
Získání EDR detekčního kontextu — process tree, ransom note lokace, postižené adresáře
Hash lookup přes SOAR (VirusTotal, MRTI, interní TI) — identifikace ransomware rodiny
Ověření, zda šifrování stále probíhá; ověření VSS stavu v logu
Eskalace na L2 ihned — ransomware = vždy eskalovat bez čekání na VT výsledek
## SOC L2
Převzetí incidentu, validace severity P1, koordinace s IR týmem
Rozhodnutí o rozsahu izolace — endpoint nebo celý segment; schválení SOAR akce
Izolace endpointu přes MDE / Bitdefender API; FW rule pro síťový segment
Izolace backup systémů — instrukce zákazníkovi pro fyzické nebo logické odpojení
First notification zákazníkovi a managementu do 15 minut
IOC extrakce a distribuce na FW, proxy, mail gateway, SIEM alerting rules
Koordinace s IR týmem — sběr forenzních artefaktů před eradikací
Průběžné aktualizace zákazníkovi každých 30 minut
## IR Team (L3)
Forenzní sběr: memory dump, $MFT, Windows event logy, Sysmon, prefetch, registry hives — PŘED eradikací
Ransomware analýza: statická (YARA, strings, imports, packer) + dynamická (sandbox)
Root cause analysis — initial access vector (phishing, exposed RDP, VPN exploit, supply chain)
Lateral movement analýza — rozsah šíření, kompromitované hosty, použité techniky
Vyhodnocení decryptor dostupnosti (NoMoreRansom.org, vendor intelligence)
Eradikace — reimaging postižených hostů (patch + hardening config aplicovat)
Ověření čistého stavu — EDR full scan, persistence check, network baseline
## Zákazník / Customer IT
Fyzická nebo logická izolace zálohovacích systémů a tape/offline médií
Reset hesel — všichni kompromitovaní uživatelé, service accounts, admin accounts
Krbtgt account reset (2× s 10h rozestupem) při kompromitaci AD
Koordinace business continuity plánu — co funguje, co je nedostupné
Schválení reimagingu vs. recovery ze zálohy pro každý zasažený host
Notifikace pojišťovny (cyber insurance) — dle pojistné smlouvy
## Management / Legal
Schválení krizového komunikačního plánu — interní i externí (zákazníci, média)
Rozhodnutí o NIS2/ZKB notifikaci regulátorovi — Legal posoudí do 24 hod od detekce
Posouzení platby výkupného — Legal, management, insurance; SOC nerozhoduje
Schválení zdrojů pro IR — externa forenzní firma, DR datacenter, cloud burst capacity

# 12. Eskalační pravidla
## L1 → L2 (okamžitě — ransomware nevyžaduje čekání na VT):
Jakýkoliv alert s klíčovými slovy: ransom, encrypt, vssadmin delete, shadow copy, bcdedit /set recoveryenabled No
Masová změna přípon souborů detekována EDR nebo SIEM (>50 souborů / min)
Uživatel reportuje nepřístupné soubory nebo ransom note
SLA: do 5 minut od alertu — ransomware je vždy P1
## L2 → L3 / IR Team:
Šifrování potvrzeno na ≥1 hostu — okamžitě
VSS nebo zálohy smazány — okamžitě
Lateral movement na ≥2 hosty detekován — okamžitě
Privilegovaný účet (domain admin, service account) kompromitován
C2 komunikace nebo data exfiltrace detekována — double extortion risk
SLA: okamžitě při P1 potvrzení, paralelně s izolací
## SOC → Zákazník:
Fyzická izolace backup systémů — SOC nemá přímý přístup
Reset hesel pro privilegované accounts a domain accounts
Reimaging postižených hostů — zákazník koordinuje s IR
Schválení business continuity aktivit
SLA: první kontakt do 15 minut od P1 potvrzení; update každých 30 minut
## Major Incident / Management escalation:
P1 ransomware — jakýkoliv potvrzený ransomware = major incident
≥3 zasažené hosty nebo kritický server (DC, ERP, DB server)
Zálohy kompromitovány nebo nedostupné
Indikace data exfiltrace (double extortion) nebo veřejná hrozba úniku dat
Pojišťovna nebo regulátor musí být notifikováni
SLA: management alert do 15 min od P1; war room otevřít do 30 min

# 18. Podmínky uzavření incidentu
Všechny zasažené hosty prošly reimagingem nebo byly prokazatelně vyčištěny (EDR full scan clean + persistence check)
IOC blokovány na FW, proxy, mail gateway, SIEM korelační rule aktivní (timestamp blokace v ticketu)
Zálohy ověřeny jako nepoškozené nebo data úspěšně obnovena ze zálohy, zákazník potvrdil
Kompromitované účty resetovány (hesla, MFA tokeny), Krbtgt reset proveden pokud bylo AD kompromitováno
VSS / Shadow Copies obnoveny nebo nová zálohovací politika implementována
Zákazník potvrdil obnovení normálního provozu všech kritických systémů
Root cause identifikován, initial access vector zdokumentován a patch/mitigace aplikována
NIS2/ZKB/GDPR notifikace posouzena a pokud povinná, pak provedena a zdokumentována
Pojišťovna notifikována (pokud cyber insurance) a claim zahájen
Lessons learned vyplněny a schváleny (povinné P1/P2 — do 5 pracovních dní od closure)
Closure schválil: SOC L3 + IR Team + SOC Service Manager + zákazník
Ticket uzavřen se všemi povinnými poli: root cause, timeline, evidence list, action items
# 19. Lessons Learned
## Co hodnotit:
MTTD — jak rychle byl ransomware detekován od okamžiku spuštění (detection gap analýza)
Účinnost containmentu — kolik hostů bylo zasaženo před izolací (blast radius)
Stav záloh — byly zálohy dostupné a nepoškozené? Pokud NE — identifikovat příčinu a remediat
Účinnost EDR/XDR — bylo šifrování zastaveno automaticky nebo jen alertováno?
SOAR automatizace — co šlo automatizovat a nešlo? Kde zpomalil manuální krok response?
Komunikace se zákazníkem — rychlost, jasnost, spokojenost; eskalace proběhla správně?
Forenzní evidence — bylo možné provést root cause analysis? Chyběly logy?
Initial access vector — phishing, exposed RDP, VPN exploit, supply chain? Preventivní opatření?
AD kompromitace — byl rozsah credential theft plně identifikován? Všechny účty resetovány?
Recovery čas — byl výsledek acceptable pro business continuity požadavky zákazníka?
## Výstup z Lessons Learned (povinný formát):
Položky s vlastníkem, popisem a deadline: [akce | odpovědná osoba | deadline]
Aktualizace detekčních pravidel v Sentinel / Splunk / FortiSIEM — nové TTPs ransomware rodiny
EDR policy update — agresivnější behavioral blocking pro ransomware vzory
SOAR playbook update — nové automatizační kroky identifikované během incidentu
Backup policy review — offline/immutable backup, 3-2-1 pravidlo, recovery test frekvence
Threat hunting query pro proaktivní vyhledávání podobných IOC nebo precursor activity
Tabletop cvičení s novým scénářem pokud odhalena procesní mezera
# 20. Doporučení
## Technická doporučení:
EDR na 100 % endpointů — MDE/XDR nebo Bitdefender GravityZone s behaviorálním enginem aktivním
Immutable / offline zálohy — 3-2-1-1-0 pravidlo: 3 kopie, 2 média, 1 offsite, 1 offline/air-gapped, 0 chyb v recovery testu
Privileged Access Workstation (PAW) pro admin účty — izolace správy od uživatelského segmentu
Network segmentace — micro-segmentace eliminuje lateral movement blast radius
MFA povinné pro VPN, RDP, admin přístupy, M365 — blokuje credential-based initial access
Attack Surface Reduction (ASR) pravidla v MDE — blokuje Office macros → powershell, wscript, mshta execution
Application whitelisting (AppLocker / WDAC) — blokuje spuštění neznámých binárek
SMB signing povinné — prevence SMB relay útoků pro lateral movement
EDR exploit protection — AMSI, Credential Guard, LSA Protection, Memory Integrity
Pravidelný vulnerability scan a patch management — RDP/VPN/ESXi CVE jsou nejčastější initial access vector
## Procesní doporučení:
Tabletop cvičení s ransomware scénářem minimálně 2× ročně — SOC + zákazník + management
Recovery test ze zálohy minimálně 1× za kvartál — ověřit RTO/RPO v reálném prostředí
IR retainer — mít smluvního IR partnera k dispozici pro P1 incidenty (FortiSIEM, CrowdStrike, externí forenzní firma)
Ransomware response plán jako součást BCP/DRP zákazníka — SOC nerozhoduje o výkupném
Onboarding nových analytiků na tomto playbooku + tabletop simulace
## Automatizace a SOAR:
Plně automatická izolace P1 endpointů bez L2 schválení — po risk review a zákazníkském souhlasu
Automatic backup isolation alert při ransomware P1 — bez čekání na manuální krok
IOC distribution pipeline: EDR → XSOAR → FW/Proxy/Sentinel/Splunk → automaticky
SOAR integration s backup systémem — status check a isolation trigger přes API
Automated MITRE ATT&CK tagging v SOAR pro každý ransomware alert
## Governance a compliance:
Kvartální review SLA adherence — MTTD, MTTA, MTTR, containment time, false positive rate
Měsíční threat intelligence brief o aktivních ransomware skupinách a jejich TTP
NIS2/ZKB readiness assessment — notifikační proces otestovat tabletopem před reálným incidentem
Cyber insurance review — pokrývá incident response náklady, data recovery, regulatorní pokuty?
Tento playbook revidovat při každé major verzi MITRE ATT&CK nebo nové ransomware kampani cílící na sektor zákazníka
