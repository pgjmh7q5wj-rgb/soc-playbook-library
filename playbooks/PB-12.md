## 1. ÚČEL

Playbook řeší detekci a reakci na úspěšné nebo neúspěšné přihlášení uživatele (zejména privilegovaného účtu) z geolokace, která je klasifikována jako blokovaná nebo riziková země. Pokrývá riziko kompromitace účtu prostřednictvím credential stuffing, pass-the-hash, phishingu nebo neoprávněného přístupu z jurisdikcí mimo povolené oblasti. Playbook existuje proto, že přihlášení z blokované geolokace je silný indikátor kompromitace účtu — zejména v kombinaci s privilegovaným přístupem nebo administrátorskou aktivitou.

---

## 2. ROZSAH POUŽITÍ

**V scope:**
- Přihlášení do Entra ID (Azure AD) / Active Directory z blokované nebo neočekávané geolokace
- Přihlášení do cloudových služeb (Microsoft 365, Azure Portal, VPN) z blokované země
- Administrátorská a privilegovaná přihlášení (Global Admin, Domain Admin, privilegované service accounts) z jakékoli neočekávané geolokace
- Logy z Entra ID Sign-in, MFA logy, Conditional Access logy
- SIEM alerty z FortiSIEM, Sentinel, Splunk korelující geolokaci s přihlašovacím eventem
- Alerty z Cortex XDR / Defender Identity (UEBA modul)

**Mimo scope:**
- Přihlášení z povolených zahraničních lokalit (zaměstnanci na dienstu v zahraničí – nutné schválení a výjimka v systému)
- VPN přihlášení, pokud je user přihlášen přes firemní VPN s exit node v povolené zemi
- Automatizované service account přístupy, pokud mají zdokumentované fixed IP adresy
- Tento playbook neřeší samotný phishing jako vektor (viz PB-01 Phishing playbook)

**Playbook se nepoužívá pokud:**
- Existuje platná výjimka geolokace pro daného uživatele (schválená SOC Managerem nebo Customer Security)
- Uživatel je přihlášen přes schválenou firemní VPN (exit node v povolené zemi) a IP je ověřena
- Alert byl již uzavřen jako FP v posledních 24 hodinách pro stejného uživatele a stejnou zemi

## 5. DETEKČNÍ LOGIKA

### Co se detekuje

Detekce přihlášení (úspěšného i neúspěšného) z IP adresy mapující se na geolokaci klasifikovanou jako:
1. Blokovaná/riziková země dle politiky zákazníka (Named Locations policy v Entra ID / SIEM blocklist)
2. Přihlášení z Tor exit node nebo anonymizačního proxy/VPN hostitele
3. Impossible Travel – dva úspěšné logony jednoho uživatele z geograficky vzdálených míst v čase kratším, než je fyzicky možné cestovat (threshold: typicky < 2 hodiny / > 500 km)
4. Privilegovaný účet (Admin, Service Account s elevated rights) přihlašující se z jakékoli neočekávané geolokace

### Proč je to podezřelé

Přihlášení z blokované geolokace indikuje: (a) kompromitaci účtu – útočník operuje z jiné jurisdikce, (b) credential stuffing útok, (c) obejití Conditional Access politiky přes Tor/proxy, nebo (d) insider threat z neschválené lokace. U privilegovaných účtů je riziko kriticky vyšší – jedno úspěšné přihlášení může vést k plné kompromitaci prostředí.

### Indikátory kompromitace (IoC)

- **Geolokace IoC**: Zdrojová IP mapující se na blokovanou zemi dle GeoIP (MaxMind / Azure / FortiSIEM GeoIP)
- **Anonymizace IoC**: IP patřící do Tor exit node databáze, known VPN/proxy hosting ASN
- **Behaviour IoC**: Přihlášení z nové geolokace pro daného uživatele (zero/first occurrence v historii 90 dnů)
- **Temporal IoC**: Přihlášení mimo pracovní hodiny (23:00–05:00 lokálního času zákazníka) z blokované geolokace
- **MFA IoC**: Úspěšné přihlášení bez MFA nebo s MFA fatigue (opakované push notifikace těsně před logon success)
- **Privilege IoC**: Přihlašující se účet má roli Global Admin / Domain Admin / Privileged Role Administrator / Exchange Admin

### Co zvyšuje severity

| Podmínka | Dopad na severity |
|---|---|
| Přihlášení úspěšné (ne jen pokus) | Zvýšit o 1 stupeň (P3 → P2) |
| Přihlášený účet je privilegovaný (admin) | Zvýšit na P2 minimum |
| Privilegovaný účet + úspěšné přihlášení + blokovaná země | Eskalovat na P1 |
| Impossible Travel detekován (2 úspěšné logony) | Zvýšit na P2 minimum |
| MFA bylo obejito nebo nebylo vyžadováno | Zvýšit o 1 stupeň |
| Přihlášení z Tor exit node nebo known malicious IP | Zvýšit na P2 minimum |
| Post-login admin aktivita detekována (role change, new user, policy change) | Eskalovat na P1 |
| Blokovaná země = riziková jurisdikce (dle EU/US sanction list) | Zvýšit na P2 minimum |

### False positive podmínky

- Uživatel má schválenou výjimku geolokace (cestování, home office v zahraničí) – ověřit v systému výjimek
- Uživatel je na dovolené v zahraničí a přihlásil se bez VPN – ověřit s HR / helpdesk ticketem
- Firemní VPN exit node je hosted v zahraniční lokalitě (mapuje se jako blokovaná země) – ověřit s IT
- Shared service account přistupující z cloud infra (Azure / AWS region) mimo povolené geolokace – ověřit s infrastrukturou
- Test/sandbox přihlášení z testovacího prostředí s VPN v jiné zemi

### SIEM pseudo-query

**Sentinel (KQL) – Geo-blocked login:**
```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| extend Country = tostring(LocationDetails.countryOrRegion)
| where Country in ("RU", "CN", "KP", "IR", "BY", "SY")  // doplnit dle politiky zákazníka
| where ResultType == "0"  // 0 = success
| project TimeGenerated, UserPrincipalName, IPAddress, Country, AppDisplayName,
          ConditionalAccessStatus, MfaDetail, UserAgent
| sort by TimeGenerated desc
```

**Sentinel (KQL) – Privileged account geo-blocked login:**
```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| extend Country = tostring(LocationDetails.countryOrRegion)
| where Country in ("RU", "CN", "KP", "IR", "BY", "SY")
| join kind=inner (
    IdentityInfo
    | where AssignedRoles has_any ("Global Administrator", "Privileged Role Administrator", "Exchange Administrator")
    | project AccountUPN, AssignedRoles
) on $left.UserPrincipalName == $right.AccountUPN
| project TimeGenerated, UserPrincipalName, AssignedRoles, IPAddress, Country, ResultType
```

**Splunk – Geo-blocked login (Entra ID / Azure AD logy):**
```spl
index=azure_ad sourcetype="azure:aad:signin"
| iplocation src_ip
| where Country IN ("Russia", "China", "North Korea", "Iran", "Belarus", "Syria")
| eval result=if(ResultType="0","SUCCESS","FAILURE")
| stats count by UserPrincipalName, src_ip, Country, result, AppDisplayName
| where result="SUCCESS"
| sort -count
```

**FortiSIEM – Geo-blocked login korelační pravidlo (pseudo):**
```
Rule: GEO_BLOCKED_LOGIN_SUCCESS
Event Type: WinAuth-Logon-Success OR AADSignIn-Success
Filter: src_geo_country IN [BLOCKED_COUNTRY_LIST]
Threshold: 1 event in 5 min
Severity: HIGH
Action: Create incident, trigger SOAR
```

**Impossible Travel – Sentinel (KQL):**
```kusto
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType == "0"
| project TimeGenerated, UserPrincipalName, IPAddress,
          Lat=toreal(LocationDetails.geoCoordinates.latitude),
          Lon=toreal(LocationDetails.geoCoordinates.longitude)
| sort by UserPrincipalName asc, TimeGenerated asc
| serialize
| extend PrevTime = prev(TimeGenerated, 1), PrevLat = prev(Lat, 1), PrevLon = prev(Lon, 1), PrevUser = prev(UserPrincipalName, 1)
| where UserPrincipalName == PrevUser
| extend TimeDiffHours = datetime_diff('hour', TimeGenerated, PrevTime)
| extend DistKm = geo_distance_2points(Lon, Lat, PrevLon, PrevLat) / 1000
| where TimeDiffHours < 2 and DistKm > 500
| project TimeGenerated, UserPrincipalName, IPAddress, TimeDiffHours, DistKm
```

## 10. REAKČNÍ KROKY

### SOC L1

1. SOC L1 přijme alert, otevře ticket v ServiceNow/Jira, zaznamená timestamp detekce.
2. SOC L1 provede enrichment zdrojové IP: GeoIP lookup, Tor/proxy/VPN flag, reputace v TI feedu (XSOAR automaticky).
3. SOC L1 ověří identitu uživatele v Entra ID / AD: role, oddělení, přiřazené privilegované role.
4. SOC L1 zkontroluje systém výjimek geolokace – pokud výjimka existuje, uzavře ticket jako FP s dokumentací.
5. SOC L1 zkontroluje VPN logy: byl uživatel přihlášen přes VPN v době incidentu? Pokud ano, uzavře jako FP.
6. SOC L1 ověří MFA logy: byl login chráněn MFA? Výsledek zaznamená do ticketu.
7. SOC L1 spustí SIEM query na historii přihlášení uživatele za 90 dní – ověří, zda geolokace byla dříve použita.
8. SOC L1 posoudí severity dle matice (sekce 7) a předá SOC L2 při P1/P2 nebo pokračuje při P3/P4.

### SOC L2

1. SOC L2 převezme ticket od L1, ověří závěry enrichmentu.
2. SOC L2 provede korelaci: Impossible Travel check, post-login aktivita v Entra Audit logs / Defender for Identity / Sentinel.
3. SOC L2 posoudí admin akce po logonu: kontrola Entra ID Audit log pro dané UPN za posledních 60 minut (role changes, policy updates, new app registrations, new user creation).
4. SOC L2 finalizuje severity a eskaluje dle pravidel (sekce 11).
5. SOC L2 odešle first notification zákazníkovi dle šablony (sekce 12).
6. SOC L2 rozhodne o containmentu a buď spustí automatický XSOAR workflow, nebo požádá zákazníka o manuální akci.
7. SOC L2 průběžně dokumentuje všechna rozhodnutí, timestamps, důkazy v ticketu.

### SOC L3 (aktivován při P1 nebo komplexním P2)

1. SOC L3 přebírá koordinaci incidentu od L2.
2. SOC L3 provede hloubkovou analýzu: všechny session tokeny uživatele, OAuth grants, přihlášení aplikací, Conditional Access bypass techniky.
3. SOC L3 koordinuje IR tým při potvrzené kompromitaci.
4. SOC L3 zajistí management escalation dle SLA.
5. SOC L3 řídí eradikaci a recovery proces.

### Zákazník (Customer IT / Customer Security)

1. Customer IT provede **disable account** kompromitovaného uživatele v AD / Entra ID ihned po pokynu od SOC.
2. Customer IT provede **revoke all sessions** v Entra ID: `Revoke-AzureADUserAllRefreshToken` nebo přes Entra ID Portal → User → Revoke sessions.
3. Customer IT provede **reset hesla** kompromitovaného účtu z bezpečného zařízení a lokace.
4. Customer IT zkontroluje a odvolá všechny OAuth aplikace přiřazené uživateli, které mohl útočník zneužít.
5. Customer Security zkontroluje Entra ID Audit log na admin akce provedené v časovém okně incidentu.
6. Customer Security ověří, zda útočník nevytvořil nové admin účty, nové app registrations nebo nezměnil CA politiky.
7. Customer IT zablokuje zdrojovou IP na firewallu / Conditional Access Named Locations (pokud dosud není blokována).
8. Customer Security potvrdí zákazníkovi SOC po provedení containmentu (email/ticket).

### IR tým (aktivován při P1 nebo potvrzené kompromitaci)

1. IR tým převezme forenzní analýzu: kompletní session timeline, Entra ID audit log export, sign-in log export.
2. IR tým analyzuje post-login akce útočníka: co bylo přečteno, změněno, staženo, vytvořeno.
3. IR tým prověří lateral movement: přístupy na další systémy z kompromitovaného účtu (SharePoint, Exchange, Teams, Azure).
4. IR tým provede root cause analysis: jak byly credentials získány (phishing, credential stuffing, leaked credentials).
5. IR tým zajistí eradikaci: odstranění všech backdoor přístupů (rogue MFA device, new admin account, app grant).
6. IR tým vypracuje IR report s timeline, dopadem, root cause a doporučeními.

### Management (P1 / velký P2)

1. SOC Manager notifikuje Management do 30 minut od P1 klasifikace.
2. Management obdrží executive update (šablona v sekci 12): typ incidentu, dotčený účet, potenciální dopad, provedené kroky.
3. Management rozhoduje o nutnosti notifikace regulátora (NIS2 / ZKB – 24h reporting window).
4. Management schvaluje případné extended downtime nebo disable kritických administrátorských účtů.

## 18. PODMÍNKY UZAVŘENÍ INCIDENTU

Incident PB-12 lze uzavřít pouze pokud jsou splněny **všechny** níže uvedené podmínky:

- [ ] Příčina přihlášení byla identifikována (kompromitace / FP / benign) a dokumentována v ticketu
- [ ] Pokud TRUE POSITIVE: containment byl proveden a potvrzen — account disable + session revoke + IP block
- [ ] Pokud TRUE POSITIVE: zákazník potvrdil provedení containmentu (email nebo ticket comment s timestamp)
- [ ] Post-login audit log byl zkontrolován a výsledek dokumentován (admin akce: ANO/NE + detail)
- [ ] Všechny backdoor přístupy identifikovány a odstraněny (rogue MFA device, app grants, nové admin účty)
- [ ] Všechny relevantní IoC (IP, UPN, session tokens) jsou zablokovány nebo zneplatněny
- [ ] Auditní evidence je kompletní v ticketu (raw logy, enrichment výstupy, komunikace se zákazníkem)
- [ ] Closure reason vyplněn: `TRUE POSITIVE` / `FALSE POSITIVE` / `BENIGN` / `UNDETERMINED`
- [ ] Pokud FALSE POSITIVE: FP důvod zdokumentován a zvážen pro tuning detekčního pravidla
- [ ] Pokud P1: IR report dokončen nebo v procesu; management informován o closure

**Closure potvrzuje:**
- P3/P4 → SOC L2
- P2 → SOC L2 (SOC Manager informován)
- P1 → SOC Manager (Management informován; IR report přiložen)

## 20. DOPORUČENÍ

### Technická

1. **Implementovat Conditional Access Named Locations policy** v Entra ID – blokovat přihlášení z definovaných zemí přímo na úrovni IdP, nejen detekovat v SIEM. Tím se eliminuje celá kategorie P2/P3 alertů preventivně.
2. **Vynutit MFA pro všechny privilegované účty** bez výjimky; zvážit FIDO2 / hardware token pro Global Admin role (odolné vůči phishing a MFA fatigue).
3. **Nasadit Privileged Identity Management (PIM)** v Entra ID – admin role aktivovány pouze JIT (Just-In-Time) na omezenou dobu, čímž se radikálně snižuje útočná plocha kompromitovaného admin účtu.
4. **Rozšířit SIEM korelaci** o detekci post-login admin akcí: korelovat geo-blocked login success s Entra Audit logem v okně 60 minut → automatický P1 trigger bez nutnosti manuálního kroku.
5. **Integrovat GeoIP databázi s aktuálními Tor exit node a VPN hosting ASN listy** do SIEM (automatická aktualizace min. 1× týdně) – snížení FP z přihlášení přes komerční VPN.
6. **Nasadit UEBA baseline** v Cortex XDR / Sentinel – behaviorální anomálie geolokace budou detekovány i v případech, kdy útočník pochází z „povolené" geolokace (kompromitace místního zařízení).

### Procesní

1. **Vytvořit a udržovat aktuální registr geolokačních výjimek** – schvalovací proces (manager + Customer Security), platnost výjimky max. 30 dní, automatická expirace. Bez registru SOC tráví čas manuálním ověřováním FP.
2. **Definovat závazný seznam blokovaných zemí** ve spolupráci se zákazníkem (Customer Security + Legal) – zdokumentovat v playbooku, aktualizovat min. 1× ročně nebo po geopolitické změně.
3. **Nastavit SOC eskalační kontakty na straně zákazníka** (Customer Security + IT) s garantovanou dostupností pro P1/P2 mimo pracovní dobu – zdokumentovat v SLA.
4. **Provádět čtvrtletní tabletop cvičení** pro scénář „kompromitovaný admin účet z blokované geolokace" – ověřit, zda zákazník zná postup, ví jak disable account a revoke sessions.

### Governance

1. **Zahrnout geo-based login monitoring do pravidelného SOC reportingu** zákazníkovi (měsíční report: počet alertů, TP/FP ratio, geolokace, trendy) – zvyšuje viditelnost rizika pro management.
2. **Zdokumentovat NIS2 / ZKB reporting workflow** pro P1 incident s dopadem na privilegovaný účet – kdo posuzuje, kdo notifikuje, v jakém čase. Nejasný proces při reálném incidentu způsobuje zpoždění a compliance riziko.
3. **Provádět roční review seznamu privilegovaných účtů** – odebírat nepotřebné admin role, auditovat service accounts s elevated rights. Zbytečné privilegované účty zvyšují útočnou plochu.
4. **Zahrnout geo-based login SLA do zákaznické smlouvy** (SLA dokument) – MTTA < 15 min pro P2, < 5 min pro P1; zákazník musí vědět, co od SOC očekávat a co je od zákazníka vyžadováno.

### Automatizace

1. **Přidat automatický post-login admin akce check** do XSOAR playbooku – při každém geo-blocked login success automaticky query Entra Audit log za posledních 60 minut; pokud admin akce nalezeny → automaticky eskalovat na P1.
2. **Automatizovat FP closure** při potvrzené výjimce nebo VPN přihlášení – XSOAR zkontroluje registr výjimek a VPN logy; pokud obojí negativní, teprve pak vytvoří plný ticket pro L1.
3. **Implementovat automatic session revoke** pro P1 bez čekání na zákazníka – definovat v SLA s zákazníkem, že SOC má právo revokovat sessions privilegovaného kompromitovaného účtu bez předchozího schválení při P1. Zdokumentovat v zákaznické smlouvě.
4. **Nastavit weekly reporting SOAR dashboard** – geo-blocked login trend, top source countries, TP/FP ratio, average MTTA/MTTR – automaticky generovaný a zasílaný zákazníkovi.
