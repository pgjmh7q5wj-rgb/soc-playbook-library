# 2. Účel playbooku
Tento playbook definuje standardizovaný postup pro detekci, triage, containment a eradikaci incidentů spojených s útoky hrubou silou (brute force) na autentizační systémy, zejména v případě, kdy útok vyústil v úspěšné přihlášení.
Pokrývá scénáře: password guessing, password spraying, credential stuffing, MFA fatigue. Klíčový eskalační trigger je kombinace neúspěšných pokusů + úspěšný login ze stejné IP nebo účtu — tento scénář znamená kompromitaci přihlašovacích údajů a musí být okamžitě řešen jako bezpečnostní incident.
Playbook zajišťuje konzistentní, auditovatelný a měřitelný response proces od detekce přes containment po uzavření incidentu.

### V scope:
Active Directory (on-prem) — SMB, LDAP, Kerberos, NTLM brute force
Microsoft Entra ID / Azure AD — cloud authentication, risky sign-in events
VPN přihlášení (Cisco, Fortinet, Palo Alto) — opakované failed authentication

Webové aplikace s vlastním autentizačním systémem (pokud logovány do SIEM)
SSH — Linux servery, network devices
Lokální přihlášení na endpoint — Windows Security Event Log 4625
### Mimo scope:
Fyzický přístup — zámky, přístupové karty, biometrické systémy bez IT integrace
Aplikace bez autentizačních logů v SIEM — nutno doplnit logovací pokrytí
Interní testování bezpečnosti (pentest) — ověřit před zahájením triage
Automatizované monitoring nástroje způsobující false lockouty — ověřit v CMDB

# 6. Detekční logika
### Primární indikátory — Brute Force bez úspěšného loginu:
≥ 10 failed login events (Event ID 4625) na jeden účet v 5 minutách ze stejné IP
≥ 5 různých účtů s failed loginy z jedné IP v 10 minutách → Password Spray (T1110.003)
Credential stuffing: failed loginy z různých IP adres, různé User-Agent strings, vysoký objem
Account lockout (Event ID 4740) na produkčním nebo privilegovaném účtu
VPN: ≥ 20 failed auth z jedné IP v 15 minutách
MFA Fatigue: ≥ 5 MFA push notifikací odmítnutých uživatelem v 30 minutách
### Kritický escalation trigger — Brute Force + Successful Login:
Event ID 4625 (failed) → 4624 (success) z TÉŽE IP nebo na TENTÝŽ účet v časovém okně 60 min → OKAMŽITĚ P2/P1
Entra ID Risky Sign-In + úspěšná autentizace → P2 bez ohledu na počet pokusů
Úspěšné přihlášení na privilegovaný účet (Domain Admin, Global Admin) po jakémkoli BF patternu → P1
Impossible travel: úspěšný login z geo-lokace vzdálené > 500 km v časovém okně < 2 hod → T1078
### SIEM korelace — Microsoft Sentinel (KQL):
// Brute Force + Successful Login — korelace
let BruteForce = SecurityEvent
| where TimeGenerated > ago(1h)
| where EventID == 4625
| summarize FailedCount = count(), Accounts = dcount(TargetUserName)
by IpAddress, bin(TimeGenerated, 5m)
| where FailedCount >= 10 or Accounts >= 5;
let SuccessfulLogin = SecurityEvent
| where TimeGenerated > ago(1h)
| where EventID == 4624
| project SuccessTime = TimeGenerated, IpAddress, Account = TargetUserName;
BruteForce
| join kind=inner (SuccessfulLogin) on IpAddress
| where SuccessTime > TimeGenerated
| project IpAddress, FailedCount, Accounts, SuccessTime, Account
| extend Severity = iff(Accounts > 5, 'P1', 'P2')
### SIEM korelace — Splunk (SPL):
index=wineventlog EventCode=4625
| bucket _time span=5m
| stats count as FailedAttempts dc(user) as UniqueAccounts by src_ip, _time
| where FailedAttempts >= 10 OR UniqueAccounts >= 5
| join type=inner src_ip [
search index=wineventlog EventCode=4624
| stats count by src_ip, user
]
| eval Risk=if(UniqueAccounts >= 5, "PasswordSpray-P1", "BruteForce-P2")
| table _time src_ip user FailedAttempts UniqueAccounts Risk
### SIEM korelace — FortiSIEM (pseudo-rule):
Rule: BRUTE_FORCE_SUCCESSFUL_LOGIN
Conditions:
Event1: WIN-SEC-4625 count(srcIpAddr) >= 10 in 5 min
Event2: WIN-SEC-4624 srcIpAddr = Event1.srcIpAddr within 60 min of Event1
Action: Trigger INCIDENT P2, notify SOC L1, XSOAR enrichment
Severity Override: P1 if targetUser in (Domain Admins, Enterprise Admins, Global Admins)
### False positive scénáře:
Pentest / Red Team aktivita — ověřit s zákazníkem a change management před triage
Misconfigured aplikace s tvrdě zakódovanými credentials — ověřit service account owner
Uživatel zapomněl heslo — korelovat s helpdesk ticketem, nízký počet pokusů
Load balancer / NAT — více uživatelů za stejnou IP (NAT pool) → zkontrolovat User-Agent a username variety
Monitoring nástroje (Nagios, SCOM) — ověřit v CMDB jako autorizovaný zdroj

# 9. Rozhodovací logika
### Decision tree — Brute Force + Successful Login:
Byl detekován úspěšný login (Event 4624) ze stejné IP nebo na stejný účet po BF patternu?
ANO → Přejít na větev KOMPROMITACE (kroky níže)
NE → P3 monitoring, watchlist IP, vyhodnotit lockout, eskalovat pouze pokud pokračuje
VĚTEV KOMPROMITACE — Byl zasažen privilegovaný účet (Domain Admin / Global Admin / Service Account)?
ANO → P1 okamžitě, aktivovat IR Team, war room, management notification, zvážit AD forest recovery
NE → P2, eskalovat na L2, pokračovat triage
Je detekováno Impossible Travel (geo-distance > 500 km v < 2 hod)?
ANO → P2/P1 dle kontextu, okamžitý reset hesla a revoke session
NE → pokračovat
Je detekován MFA Fatigue (>= 5 odmítnutých push) a následně úspěšná MFA auth?
ANO → P2, útočník přiměl uživatele potvrdit MFA, okamžitý reset + zablokování
NE → pokračovat
Je detekována post-login aktivita: nové inbox rules, delegace, MFA device registrace, OAuth consent?
ANO → P1/P2, persistence mechanismus identifikován, IR Team, forenzní analýza
NE → P2 s monitoringem
Je detekován lateral movement z kompromitovaného účtu?
ANO → P1, rozšířit containment na zasažené systémy, IR Team
NE → P2 izolovaný na jeden účet, standardní workflow
# 10. Reakční kroky
## SOC L1
Příjem alertu, ověření účtu v AD/Entra ID, vytvoření ticketu
Identifikace typu útoku: BF vs. spray vs. credential stuffing vs. MFA fatigue
KRITICKÝ KROK: ověřit přítomnost Event ID 4624 (úspěšný login) po BF patternu
Obohacení zdrojové IP: VirusTotal, geolokace, ASN, known BF botnet
Ověřit privilegia cílového účtu — eskalovat okamžitě pokud admin
Zdokumentovat detekční kontext do ticketu: čas, IP, účet, počet pokusů, výsledek
Eskalovat na L2 pokud: úspěšný login, privilegovaný účet, impossible travel, MFA fatigue+success
## SOC L2
Převzetí ticketu, validace severity, deep dive do post-login aktivity
Entra ID: zkontrolovat risky sign-in, audit log, app registrations, MFA enrollments
Rozhodnutí o okamžitém resetu hesla a revokaci session tokenů (P2/P1)
Blokace útočné IP na FW a Conditional Access Named Locations
Notifikace zákazníka — first notification email s detailem
IOC distribuce: IP na FW, proxy, Sentinel TI indicator ingestion
Koordinace s IR týmem při P1 nebo lateral movement potvrzení
## IR Team (L3)
Forenzní analýza: post-login aktivita, persistence mechanismy (inbox rules, OAuth apps, MFA devices)
AD/Entra ID analýza: všechny přihlášení kompromitovaného účtu, skupiny, delegace, GPO změny
Credential dump detekce: ověřit, zda útočník extrahoval další credentials
Lateral movement mapping: jaké systémy byly přístupné z kompromitovaného účtu
Root cause analysis: jak útočník získal credentials (phishing, credential leak, BF)
Eradikace: reset všech hesel, revoke tokens, odstranit persistence mechanismy
## Zákazník / Customer Security
Reset hesla pro kompromitované účty — provést OKAMŽITĚ bez čekání na SOC instrukce pokud P1
Revokace Entra ID session tokenů (Revoke-AzureADUserAllRefreshToken)
Zkontrolovat a odstranit: podezřelá inbox pravidla, delegace, OAuth app consenty
MFA re-enrollment na čistém zařízení pokud MFA fatigue nebo MFA device kompromitace
Ověřit Conditional Access policy — zajistit MFA pro privilegované přístupy
Informovat zasaženého uživatele — instrukce pro bezpečné chování

# 12. Eskalační pravidla
### L1 → L2:
Detekován úspěšný login (Event 4624) po BF patternu ze stejné IP nebo na stejný účet
Cílový účet je privilegovaný (Domain Admin, Global Admin, Service Account)
Impossible Travel detekováno po úspěšném loginu
MFA Fatigue + úspěšná autentizace
IP identifikována jako known malicious (VT score ≥ 5, known BF botnet)
Entra ID Risk Level: High nebo Critical
SLA: do 15 minut od alertu (P2/P1), do 30 minut (P3 s escalation trigger)
### L2 → L3 / IR Team:
Kompromitace privilegovaného účtu potvrzena nebo lateral movement detekován
Post-login persistence mechanismus identifikován (inbox rules, OAuth app, MFA device)
AD/Entra ID tenant-wide kompromitace signály (více zasažených účtů, GPO změny)
Credential dumping po úspěšném loginu detekován EDR
SLA: okamžitě při P1, do 30 minut při P2 s IR triggerem
### SOC → Zákazník (okamžitá akce):
Reset hesla kompromitovaného účtu — SOC nemůže provést sám bez zákaznického oprávnění
Revokace session tokenů v Entra ID (Revoke-AzureADUserAllRefreshToken)
MFA re-enrollment — vyžaduje spolupráci uživatele
Ověření a odstranění persistence mechanismů (inbox pravidla, OAuth consent)
SLA: do 15 minut od P2/P1 potvrzení
### Major Incident / Management escalation:
Kompromitace privilegovaného účtu (Domain Admin, Global Admin, CISO, C-level)
AD nebo Entra ID tenant-wide útok — více než 5 kompromitovaných účtů
Útok koordinovaný s ransomware nebo APT indikátory
Regulatorní notifikační povinnost (NIS2, GDPR data breach)

# 18. Podmínky uzavření incidentu
### Povinné podmínky pro closure — Brute Force bez kompromitace (P3/P4):
Útočná IP blokována na FW nebo CA Named Locations
Žádné nové BF pokusy z téže IP v posledních 2 hodinách
Cílový účet: stav ověřen, heslo v pořádku, MFA aktivní
Ticket obsahuje: typ útoku, počet pokusů, zdrojová IP, výsledek (no compromise), evidence
### Povinné podmínky pro closure — BF + Successful Login (P2/P1):
Heslo kompromitovaného účtu bylo resetováno — potvrzení zákazníka v ticketu
Entra ID session tokeny revokány — log evidence s timestampem
Útočná IP blokována na FW a CA — ověřeno
Post-login aktivita prověřena: žádné inbox rules, delegace, MFA devices nebo OAuth app consenty nezůstaly
MFA re-enrollment dokončen pokud byl zasažen MFA fatigem
Zákazník potvrdil obnovení normálního přístupu pro legitimního uživatele
IR Team (pokud aktivován) potvrdil eradikaci a closed IR report
Lessons learned zdokumentovány v ticketu
### Kdo potvrzuje closure:
P3/P4: SOC L2 — closure po ověření podmínek
P2: SOC L2 + zákazník security confirmation
P1: IR Team + SOC Service Manager + zákazník CISO confirmation
# 19. Lessons Learned
### Co sledovat po každém P1/P2 incidentu:
Jak útočník získal credentials? Phishing, credential leak (dark web), slabé heslo, insider?
Bylo MFA aktivní? Pokud ne → eskalovat na zákazníka jako CRITICAL gap
Jak dlouho probíhal BF útok před detekcí? Pokud > 30 min → tuning SIEM threshold
Byl brute force součástí koordinovaného útoku (ransomware, APT)? → rozšíření triage
Proběhl post-login inbox pravidla nebo OAuth consent útok? → doporučit Microsoft Defender for Cloud Apps
### Typické slabiny detekce:
Legacy authentication protokoly (Basic Auth) bez MFA → slepé místo pro credential stuffing
Chybějící logovací pokrytí VPN a SSH do SIEM → BF na perimetru nezachycen
Příliš vysoký BF threshold (> 50 pokusů) → útoky s nízkým tempem unikají
Absence impossible travel pravidla → kompromitace z jiné geo-lokace nezachycena
Chybějící Entra ID Sign-In log integrace do SIEM → risky sign-in nekoreluje s BF
### Doporučené akce po incidentu:
Owner: SIEM Engineer — zkontrolovat a snížit BF threshold na ≤ 10 pokusů / 5 min
Owner: Zákazník Security — vynutit MFA pro všechny privilegované účty (CA policy)
Owner: Zákazník IT — zakázat Legacy Authentication protokoly (Basic Auth, NTLM)
Owner: SOC L3 — ověřit pokrytí VPN a SSH logů v SIEM, doplnit chybějící zdroje
Owner: SOAR Engineer — verifikovat BF+login korelaci a SLA 2 min pro auto-eskalaci
# 20. Doporučení
### Technická doporučení:
Vynutit MFA (FIDO2 / Microsoft Authenticator) pro VŠECHNY uživatele — zejména privilegované účty
Zakázat Legacy Authentication v Entra ID (Basic Auth, POP, IMAP bez MFA) — Conditional Access policy
Nasadit Microsoft Entra ID Protection — automatická risky sign-in detekce a remediation
Implementovat Conditional Access: MFA + Compliant Device + Named Location pro VPN a admin přístupy
Nasadit Microsoft Defender for Cloud Apps — detekce impossible travel, OAuth consent abuse, inbox manipulation
Integrovat Entra ID Sign-In logy do Sentinel/Splunk — korelace s on-prem AD autentizacemi
Snížit BF lockout threshold: ≤ 10 pokusů → lockout, nebo progresivní delay (smart lockout)
### Procesní doporučení:
Zavést pravidelný review privilegovaných účtů (Quarterly Access Review) — PIM/PAM
Implementovat Privileged Identity Management (Entra ID PIM) — just-in-time admin přístup
Provádět credential breach monitoring (Have I Been Pwned API, dark web feeds) — měsíčně
Tabletop cvičení: scénář BF + MFA Fatigue + úspěšná kompromitace DA — 1× ročně
### Governance doporučení:
Definovat Password Policy: minimálně 16 znaků, bez complexity requirements (NIST 800-63B), bez expirací
Zavést povinné MFA re-enrollment po jakékoli kompromitaci nebo suspicious MFA activity
Dokumentovat všechny service accounts s autentizačními tokeny — rotace každých 90 dní
### Automatizace (SOAR):
Automatizovat: BF+login korelaci (SOAR auto-escalate P2 do 2 min od detekce)
Automatizovat: IP blokaci na FW a CA po L2 schválení — cíl: < 15 min od P2 potvrzení
Automatizovat: Entra ID risky sign-in → SOAR enrichment + ticket creation
Zavést: User risk score tracking — accumulate risk events per user pro dlouhodobé sledování
