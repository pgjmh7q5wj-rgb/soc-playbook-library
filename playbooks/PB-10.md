# 2. Účel playbooku
Tento playbook definuje standardizovaný postup pro detekci, triage, containment a eradikaci neautorizovaného nebo zneužitého využití cloudových zdrojů v prostředí zákazníka. Pokrývá scénáře od krádeže cloudových přihlašovacích údajů a spinování neoprávněných výpočetních instancí přes cryptomining a data staging až po exfiltraci do cizích cloudových tenantů.
Playbook existuje protože kompromitace cloudového prostředí (Azure, AWS, GCP, M365, SaaS) může vést k masivním finančním škodám (neautorizované náklady), úniku dat s GDPR/NIS2 dopady a přetrvávající persistenci útočníka mimo dosah tradičního EDR.
# 3. Rozsah použití
## V scope:
Neautorizované vytváření cloudových VM, kontejnerů, storage účtů, API klíčů
Cryptomining prostřednictvím ukradených cloudových přihlašovacích údajů
Data staging / exfiltrace do cizích cloudových tenantů (T1537)
Neautorizované změny cloudové infrastruktury (T1578) — security group rules, IAM policies
Anomální SaaS aktivita — masové stahování dat z SharePointu, OneDrive, Google Drive
Kompromitace service principals, API klíčů, OAuth tokenů
Resource abuse v M365, Azure, AWS, GCP, Salesforce, Workday
## Mimo scope:
Autorizované DevOps/CloudOps operace v rámci schváleného change ticketu
Plánované škálování infrastruktury (ověřit v CMDB / change management)
Interní pohyb dat v rámci stejného tenantu zákazníka bez exfiltrace
## Playbook se nepoužívá pokud:
Aktivita je kryta schváleným change ticketem s odpovídajícím business justification
Jedná se o plánovaný pentest / red team s předchozím souhlasem zákazníka (musí být doloženo)

# 6. Detekční logika
## Co se detekuje
Neautorizované nebo anomální operace s cloudovými zdroji pocházející z kompromitovaného účtu, ukradeného API klíče nebo OAuth tokenu. Detekce probíhá na vrstvě identity (kdo), infrastruktury (co bylo vytvořeno/změněno) a sítě (kam jdou data).
## Primární detekční scénáře

| Scénář | Indikátor | Severity |
|---|---|---|
| Cryptomining | GPU/compute instance vytvořena + DNS dotazy na mining pool (xmrig, moneroocean, pool.minergate) | P2 |
| Data exfiltrace do cizího tenantu | T1537 — storage blob copy / S3 sync na external account ID | P1 |
| Kompromitace service principal | Neobvyklé API volání ze service principal v neobvyklý čas / z nové IP | P2 |
| IAM privilege escalation | Přidání Owner/Admin role k neznámému účtu nebo service principal | P1 |
| Masový SharePoint download | Uživatel stáhl > 500 souborů za 1 hodinu nebo > 5 GB | P2 |
| Neautorizovaný VM deployment | > 5 VM za 30 minut z účtu bez předchozí takovéto aktivity | P2 |
| OAuth consent phishing | Nový OAuth app grant se širokými permissions (Mail.Read, Files.ReadWrite.All) | P2 |
| Security group / FW rule změna | Inbound allow all (0.0.0.0/0) přidáno mimo change window | P2 |

## Co zvyšuje severity
Potvrzená exfiltrace dat do cizího tenantu → P1 okamžitě
Zasažen privilegovaný účet (Global Admin, Owner, Root) → P1
Aktivita probíhá v době nepřítomnosti uživatele (OOO, dovolená, víkend) → zvýšit o 1 stupeň
Cílová destinace je na TI blacklistu → P1
GDPR / regulovaná data (PII, zdravotní, finanční) jsou dotčena → P1 + GDPR proces
Paralelní exfiltrace + IAM změna + nový VM deployment (multi-vector attack) → P1
## False positive podmínky
Schválený DevOps pipeline deployment (ověřit v CI/CD logu a change ticketu)
Plánované škálování (auto-scaling events) — ověřit s Cloud Ops týmem zákazníka
Legitimní IT admin operace v rámci schváleného change ticketu
Backup / DR operace zákazníka na sekundární cloudové úložiště (ověřit whitelist)
## SIEM pseudo-query — Neautorizovaný VM deployment (Sentinel KQL):
AzureActivity
| where OperationNameValue == "Microsoft.Compute/virtualMachines/write"
| where ActivityStatusValue == "Success"
| summarize vm_count=count() by Caller, bin(TimeGenerated, 30m)
| where vm_count > 5
| join kind=leftouter (SigninLogs | summarize by UserPrincipalName, IPAddress) on $left.Caller == $right.UserPrincipalName
## SIEM pseudo-query — Cryptomining DNS (Splunk):
index=dns query=* 
| lookup mining_pools query AS query OUTPUT is_mining 
| where is_mining=true 
| stats count by src_ip, query, _time 
| join src_ip [search index=cloud_assets] 
| table _time, src_ip, hostname, cloud_account, query
# 7. Severity model

| Priorita | Název | Technické podmínky | Business Impact | Eskalace do |
|---|---|---|---|---|
| P1 Critical | Kritický cloud incident | Potvrzená exfiltrace do cizího tenantu | IAM privilege escalation na Owner/Root | Paralelní multi-vector útok | GDPR data dotčena | Únik regulovaných dat, finanční škoda, reputační dopad, notifikační povinnost | Okamžitě → SOC L3 + IR + Management + Legal |
| P2 High | Aktivní cloud abuse | Cryptomining VM | Neautorizovaný VM deployment | Kompromitace service principal | IAM změna mimo change window | Masový SharePoint download | Neautorizované náklady, potenciální únik dat, persistence útočníka | Do 15 min → SOC L2 |
| P3 Medium | Podezřelá cloud aktivita | Neobvyklé API volání v neobvyklý čas | Nový OAuth app grant | Přihlášení z nové geolokace bez MFA challange | Potenciální kompromitace účtu, vyšetřování nutné | Do 1 hod → SOC L2 |
| P4 Low | Cloud anomálie k monitoringu | Mírná volumetrická anomálie | Neznámý user agent | Ojedinělý alert bez korelace | Nízký okamžitý dopad, preventivní monitoring | Do 4 hod → SOC L1 monitoring |

Alert přijat
│
├─ FP? (schválený change ticket / autorizovaný deployment)

│
└─ TP potvrzeno
     │
     ├─ Aktivní exfiltrace do cizího tenantu?
     │    └─ ANO → P1, okamžitá eskalace L3 + IR + Management + Legal
     │
     ├─ IAM privilege escalation (Owner / Root / Global Admin)?
     │    └─ ANO → P1, okamžitá eskalace, disable účtu
     │
     ├─ Cryptomining VM / Resource Hijacking?
     │    └─ ANO → P2, eskalace L2, terminate instance, revoke credentials
     │
     ├─ Kompromitace service principal / API klíče?
     │    └─ ANO → P2, revoke API key, audit všech operací za posledních 30 dní
     │
     ├─ Masový SharePoint / OneDrive download?
     │    └─ ANO → P2, disable účtu, ověřit obsah stažených souborů
     │
     └─ Podezřelá aktivita bez přímého dopadu
          └─ P3/P4 → monitoring, notify zákazník, 24h revisit
# 10. Reakční kroky
## SOC
SOC L1: Ticket vytvořen, alert klasifikován, SOAR enrichment spuštěn automaticky.
SOC L1: FP validace v CMDB — ověřit schválené change tickety za posledních 48 hodin.
SOC L2: Severity assessment, eskalace dle matice, zákazník notifikován při P1/P2.
SOC L2: Korelace s PB-05 (Compromised Account) — ověřit zda je dotčený účet kompromitován na endpointu.
SOC L2: Předat zákazníkovi seznam konkrétních akcí potřebných pro containment (viz níže).
SOC L3 (při P1): Koordinace IR týmu, forenzní zajištění cloudových logů, komunikace s managementem.
## Zákazník (Customer IT / Cloud Ops / Security)
Zákazník okamžitě zakáže dotčený uživatelský účet v Entra ID / AWS IAM / GCP IAM.
Zákazník revokuje všechny aktivní session tokeny, refresh tokeny a API klíče dotčeného identity.
Zákazník terminuje neautorizované VM instance, storage účty a další neautorizované resources.
Zákazník resetuje heslo + vynutí re-enrollment MFA pro dotčený účet.
Zákazník audituje IAM role assignments za posledních 30 dní — odstraní neautorizované granty.
Zákazník zkontroluje a odstraní neautorizované OAuth app consents v Entra ID.
Zákazník provede audit cloudových nákladů — identifikuje a zastaví neautorizované billing položky.
Zákazník NEODSTRAŇUJE cloudové logy (audit trail) před souhlasem SOC / IR týmu.
## IR tým (aktivace při P1 nebo komplexním P2)
Forenzní zajištění cloudových audit logů (Azure Activity Log, CloudTrail, GCP Audit) — export a záloha.
Timeline rekonstrukce — kdy byl účet / API klíč kompromitován (ověřit historii přihlášení).
Analýza rozsahu dotčených dat — jaká data byla přístupná, stažena nebo přenesena.
Root cause analysis — jak útočník získal přístup (phishing, credential stuffing, insider, supply chain).
Persistence check — hledání backdoorů: nové service principals, federated identity providers, AWS roles.
## Management (P1 / major incident)
SOC Manager informuje management do 2 hodin od P1 klasifikace.
Pokud jsou dotčena GDPR data — Legal / DPO zahájen 72h GDPR incident response proces.
Pokud jsou dotčeny finanční / cloudové náklady zákazníka — eskalace k zákaznickému managementu.

# 18. Podmínky uzavření incidentu
Incident lze uzavřít pouze pokud jsou splněny VŠECHNY následující podmínky:
Containment byl proveden a potvrzen zákazníkem (disable účtu, revoke tokenů, terminace neautorizovaných zdrojů)
Root cause byl identifikován nebo dokumentován jako 'neidentifikován' s odůvodněním
Všechny neautorizované cloudové zdroje byly odstraněny nebo izolovány
IAM role assignments a OAuth consents byly auditovány a neoprávněné položky odstraněny
Zákazník potvrdil closure (email / ticket update)
Auditní evidence je kompletní v ticketu (viz sekce 14)
GDPR posouzení proběhlo — výsledek dokumentován (notifikace nutná ANO/NE)
Closure reason je vyplněn: TRUE POSITIVE / FALSE POSITIVE / BENIGN / UNDETERMINED
**Closure potvrzuje: SOC L2 minimum. P1 → SOC Manager musí schválit.**

# 20. Doporučení
## Technická doporučení
Nasadit Microsoft Defender for Cloud (nebo AWS GuardDuty / GCP Security Command Center) s alertingem na anomální resource creation
Implementovat CASB (Defender for Cloud Apps / Netskope) pro monitoring SaaS exfiltrace a OAuth consent
Vynutit MFA pro všechna cloudová přihlášení — zejména privilegované účty a service principals
Implementovat Conditional Access policy: blokovat přihlášení z anonymizátorů (Tor, VPN bez firemní identity)
Rotace API klíčů a service principal credentials minimálně každých 90 dní, ideálně managed identity
Aktivovat Azure Cost Alerts / AWS Budgets pro detekci anomálního nárůstu nákladů
Implementovat Just-In-Time (JIT) přístup pro privilegované cloudové operace (Azure PIM, AWS IAM Access Analyzer)
## Procesní doporučení
Zákazník musí mít aktuální inventář service principals a API klíčů — revize každých 90 dní
Definovat whitelist autorizovaných cloudových destinací pro cross-tenant data transfer
Zavést offboarding checklist pro cloudové přístupy — revoke při odchodu zaměstnance do 24 hodin
Pravidelný cloud access review (IAM review) minimálně kvartálně
## Automatizace
SOAR playbook pro automatické terminování neautorizovaných VM instancí při P2 detekci (s L2 potvrzením)
Automatický alert při vytvoření nového service principal nebo API klíče mimo schválený CI/CD pipeline
Integrace Cost Management alertů do SIEM pro korelaci s bezpečnostními incidenty
