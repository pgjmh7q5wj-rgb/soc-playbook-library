## 1. UCEL

Playbook resi detekci podezrele odchozi komunikace z endpointu, serveru nebo cloud workloadu na potencialni Command-and-Control infrastrukturu. Pokryva riziko aktivni kompromitace, malware beaconingu, RAT komunikace, loader/stager aktivity a mozne exfiltrace pres C2 kanal. Playbook existuje proto, aby SOC rychle odlisil benigni outbound provoz od aktivniho rizeni kompromitovaneho systemu a spustil auditovatelny response proces.

## 2. ROZSAH POUZITI

**V scope:**
- FortiSIEM alerty na suspicious outbound, C2 beaconing, periodic connections, malware domain nebo known malicious IP.
- Microsoft Defender for Endpoint alerty typu `Command and control communication`, `Suspicious network connection`, `Possible C2 activity`, `Malware with network behavior`.
- Bitdefender GravityZone alerty na malware/RAT/trojan s odchozi komunikaci.
- XDR korelace endpoint process + network connection + threat intelligence hit.
- DNS, proxy, firewall, NetFlow a EDR telemetry pro potvrzeni C2 patternu.
- Cloud workloady a servery komunikujici na novou, neobvyklou, anonymizacni nebo malicious destinaci.
- Podezrela komunikace pres HTTP/S, DNS, TLS, uncommon ports, cloud storage nebo legitimate SaaS abuse.

**Mimo scope:**
- Cista datova exfiltrace bez C2 patternu; pouzit Data Exfiltration playbook.
- Phishing delivery bez endpoint execution; pouzit Phishing playbook.
- Bezna cloud synchronizace se schvalenymi tenanty a dokumentovanym business ownerem.
- Sitovy sum bez identifikovatelneho zdrojoveho hosta, uzivatele nebo procesu.
- Penetracni test nebo red-team aktivita, pokud existuje schvaleny testovaci zaznam.

**Playbook se nepouziva pokud:**
- Destinace je na schvalenem zakaznickem allowlistu a existuje platne business zduvodneni.
- EDR/XDR potvrzuje benigni proces a SIEM alert vznikl pouze kvuli chybne reputaci domeny.
- Alert je duplicitni k jiz otevrenemu incidentu pro stejny host, proces, destinaci a casove okno.
- Neni dostupny minimalni kontext `source host + destination + timestamp`; SOC L1 zalozi misto incidentu pozadavek na doplneni logu.

## 5. DETEKCNI LOGIKA

### Co se detekuje

Detekuje se odchozi komunikace z interniho nebo cloud aktiva, ktera odpovida jednomu nebo vice C2 indikatorum:

1. Opakovana periodicka komunikace na stejnou IP/domenu s nizkym objemem dat a stabilnim intervalem.
2. Spojeni na domenu nebo IP s malicious reputaci, malware campaign vazbou nebo nizkym TI skore.
3. DNS dotazy na DGA-like domeny, dlouhe subdomeny, vysoky NXDOMAIN pomer nebo DNS tunneling pattern.
4. HTTP/S komunikace s neobvyklym User-Agentem, URI strukturou, SNI/Host mismatch nebo newly registered domain.
5. Podezrely proces na endpointu generuje outbound spojeni mimo bezny baseline daneho hosta.
6. Cloud workload komunikuje na anonymizacni sluzbu, TOR, bulletproof hosting, neznamy ASN nebo neschvaleny externi tenant.
7. C2 komunikace je kombinovana s malware execution, persistence, credential theft, lateral movement nebo exfiltracnim objemem.

### Proc je to podezrele

C2 beaconing znamena, ze zasazene aktivum muze byt vzdalene rizene utocnikem. Pravidelna outbound komunikace na neobvyklou nebo malicious destinaci je typicka pro RAT, loader, stealer, botnet agenta nebo ransomware precursor. Pokud je C2 potvrzene spolecne s procesovou telemetrii z EDR/XDR, nejde pouze o sitovou anomalii, ale o pravdepodobnou aktivni kompromitaci vyzadujici containment.

### Indikatory kompromitace (IoC)

- **Network IoC:** opakovane spojeni `src_host -> dst_ip:dst_port` v pravidelnem intervalu.
- **Domain IoC:** newly registered domain, DGA-like domena, domena s malicious reputaci.
- **DNS IoC:** vysoky pocet NXDOMAIN, dlouhe subdomeny, vysoka entropie labelu, neobvykly query volume.
- **HTTP/S IoC:** neobvykly User-Agent, opakovany callback URI pattern, SNI/Host mismatch.
- **TLS IoC:** podezrely certificate issuer, self-signed cert, neobvykly JA3/JA4 fingerprint, pokud je dostupny.
- **Endpoint IoC:** podezrely proces, parent-child process anomaly, unsigned binary, script interpreter s network connection.
- **File IoC:** hash malware/loader/RAT, dropper ve user-writable path, archive nebo staging soubor.
- **Identity IoC:** komunikaci spousti ucet s privilegii nebo service account bez dokumentovane potreby outbound provozu.
- **Cloud IoC:** cloud VM/container/workload komunikuje mimo schvalene regiony, tenants nebo service endpoints.
- **Exfil IoC:** outbound objem nebo pattern odpovida exfiltraci pres C2 kanal.

### Co zvysuje severity

| Podminka | Dopad na severity |
|---|---|
| EDR/XDR potvrzuje proces generujici C2 komunikaci | Zvysit na P2 minimum |
| C2 komunikace je aktivni v dobe triage | Eskalovat na P1 kandidat |
| Zasazene aktivum je server, domain controller, jump server nebo cloud production workload | Zvysit o 1 stupen |
| C2 destinace je known malicious dle TI feedu | Zvysit na P2 minimum |
| Je viditelny lateral movement, credential theft, persistence nebo defense evasion | Eskalovat na P1 |
| Outbound objem indikuje exfiltraci dat | Eskalovat na P1 a navazat Data Exfiltration workflow |
| Zdrojovy ucet je privilegovany nebo service account | Zvysit o 1 stupen |
| Detekce je pouze sitova, bez EDR/process kontextu | Zachovat P3/P2 podle reputace destinace a aktiva |

### False positive podminky

- Schvaleny bezpecnostni nastroj, EDR agent, vulnerability scanner nebo monitoring platforma komunikuje na vendor cloud.
- Legitimni cloud/SaaS sluzba pouziva pravidelne heartbeat spojeni.
- CDN, update server nebo telemetry endpoint je chybne klasifikovany TI feedem.
- Vyvojarsky, CI/CD nebo cloud workload komunikuje na externi API podle schvaleneho designu.
- DNS pattern odpovida legitimnimu CDN/load-balancing mechanismu, ne DGA.
- Penetracni test/red-team aktivita ma platne schvaleni a casove okno.
- NAT nebo proxy agregace maskuje skutecny zdroj; nutno dohledat puvodni host pres firewall/DHCP/proxy logy.

### SIEM pseudo-query

**FortiSIEM pseudo-query - periodic outbound beacon:**

```sql
SELECT
  srcIp,
  hostName,
  user,
  destIp,
  destDomain,
  destPort,
  protocol,
  COUNT(*) AS connection_count,
  MIN(eventTime) AS first_seen,
  MAX(eventTime) AS last_seen,
  AVG(bytesOut) AS avg_bytes_out
FROM events
WHERE eventTime >= NOW() - 24h
  AND direction = 'outbound'
  AND action IN ('allow','permitted','success')
  AND destIp NOT IN (customer_allowlisted_ips)
  AND destDomain NOT IN (customer_allowlisted_domains)
GROUP BY srcIp, hostName, user, destIp, destDomain, destPort, protocol
HAVING connection_count >= 20
   AND avg_bytes_out < 50000
   AND is_regular_interval(first_seen, last_seen, connection_count) = true
```

**Microsoft Defender Advanced Hunting - process with suspicious outbound connection:**

```kusto
DeviceNetworkEvents
| where Timestamp > ago(24h)
| where RemoteUrl !in (dynamic(["<customer-approved-domain>"]))
| where RemoteIPType == "Public"
| summarize
    ConnectionCount=count(),
    FirstSeen=min(Timestamp),
    LastSeen=max(Timestamp),
    RemotePorts=make_set(RemotePort),
    InitiatingProcesses=make_set(InitiatingProcessFileName),
    Users=make_set(InitiatingProcessAccountName)
  by DeviceName, RemoteIP, RemoteUrl
| where ConnectionCount >= 20
| join kind=leftouter (
    DeviceProcessEvents
    | where Timestamp > ago(24h)
    | project DeviceName, InitiatingProcessFileName=FileName, ProcessCommandLine, SHA256, InitiatingProcessParentFileName
) on DeviceName
```

**DNS pseudo-query - DGA / DNS tunneling candidate:**

```sql
SELECT
  clientIp,
  queryName,
  COUNT(*) AS query_count,
  COUNT_DISTINCT(queryName) AS unique_queries,
  SUM(CASE WHEN responseCode = 'NXDOMAIN' THEN 1 ELSE 0 END) AS nxdomain_count
FROM dns_logs
WHERE eventTime >= NOW() - 6h
GROUP BY clientIp, queryName
HAVING query_count >= 100
   OR nxdomain_count / query_count > 0.5
   OR avg_domain_entropy(queryName) > customer_threshold
   OR avg_label_length(queryName) > 40
```

**XDR/SIEM korelacni pravidlo - C2 confidence scoring:**

```text
IF
  outbound_connection_to_rare_or_malicious_destination = true
  AND (
      EDR_process_context_available = true
      OR DNS_DGA_pattern = true
      OR TI_reputation_score >= high
      OR repeated_beacon_interval = true
  )
THEN
  create_incident("PB-08 C2 Beacon / Suspicious Outbound")
  set_default_severity("P2")
  attach_entities(src_host, user, process, hash, dest_ip, dest_domain, dest_port)
  run_xsoar_enrichment(ip_domain_hash_reputation, asset_criticality, 24h_timeline)
```

## 6. SEVERITY MODEL

| Priorita | Business impact | Technicke podminky | Eskalace do |
|---|---|---|---|
| **P1 Critical** | Aktivni kompromitace s rizikem sireni, lateralniho pohybu, ransomware precursoru nebo exfiltrace dat | Potvrzene aktivni C2 + lateral movement / credential theft / exfiltrace / produkcni server / cloud production workload / domain controller | Okamzite -> SOC L3 + IR tym + SOC Service Manager + Customer Security + Management |
| **P2 High** | Pravdepodobna kompromitace endpointu, serveru nebo workloadu bez potvrzeneho plosneho dopadu | EDR/XDR potvrzuje proces generujici C2 beacon; FortiSIEM korelace C2 + TI hit; malware/RAT/trojan s outbound komunikaci; known malicious destination | Do 30 min -> SOC L2; zakaznik notifikovan do 1 h |
| **P3 Medium** | Podezrela outbound aktivita bez potvrzene kompromitace | Periodicky outbound pattern, suspicious DNS/proxy aktivita, rare destination, newly registered domain, ale bez EDR potvrzeni malware procesu | Do 2 h -> SOC L1/L2 triage |
| **P4 Low** | Pravdepodobny false positive nebo tuning event | Schvaleny vendor endpoint, bezpecnostni nastroj, cloud telemetry, znamy SaaS heartbeat, TI reputace nejednoznacna | Do 8 h -> SOC L1 monitoring / closure s duvodem |

**Nutno overit:** zakaznicke SLA odchylky, kriticka aktiva, povolene cloud/SaaS destinace a schvalene automaticke containment akce.

## 9. REAKCNI KROKY

### SOC

1. SOC L1 zalozi nebo overi ticket a zaznamena alert timestamp, source host, destination, port, protokol a trigger.
2. SOC L1 spusti XSOAR enrichment pro IP/domenu/hash a vlozi vysledek do ticketu.
3. SOC L1 overi allowlist, vendor/cloud legitimitu a duplicitu incidentu.
4. SOC L1 vytahne EDR/XDR kontext: proces, parent process, command line, hash, user, device timeline.
5. SOC L2 provede korelaci FortiSIEM + firewall + DNS + proxy + EDR za 24 h a 7 dni.
6. SOC L2 stanovi severity a rozhodne, zda jde o FP, P3 monitoring, P2 incident nebo P1 major incident kandidat.
7. SOC L2 pri P1/P2 pripravi containment request: host isolation, IOC block, DNS/proxy block, pripadne cloud egress block.
8. SOC L2 dokumentuje rozhodnuti, dukazy, timestampy a ownera dalsich kroku.

### Zakaznik

1. Customer IT potvrdi kriticnost zasazeneho aktiva a business ownera.
2. Customer IT provede izolaci endpointu/serveru pres Defender, Bitdefender nebo jiny EDR, pokud SOC L2/L3 pozada o containment.
3. Customer IT zablokuje potvrzenou C2 IP/domenu/URL na firewallu, proxy nebo DNS vrstve.
4. Customer IT overi, zda komunikace nepatri schvalene aplikaci, integraci nebo cloud workloadu.
5. Customer Security schvaluje containment akce s dopadem na produkci, pokud nejsou predem schvalene pro P1.
6. Customer IT zajisti export relevantnich logu, pokud SOC nema primy pristup.
7. Customer Security potvrdi v ticketu provedeni izolace/blokace vcetne timestampu.

### IR tym

1. IR tym se aktivuje pri P1 nebo pri potvrzenem malware/RAT/C2 na kritickem aktivu.
2. IR tym provede forenzni sber: memory/process artefakty, sitova spojeni, podezrele soubory, persistence artefakty.
3. IR tym analyzuje malware/RAT/loader chovani a urci root cause.
4. IR tym overi lateral movement, credential theft a dalsi zasazene hosty.
5. IR tym doporuci eradikaci: odstraneni persistence, reimage, credential reset, patching, IOC blokace.
6. IR tym preda SOC L2/SOC L3 overene IOC pro rozsireny hunting.

### Management

1. SOC Service Manager informuje management pri P1 nebo pri P2 s dopadem na kritickou sluzbu.
2. Management dostava stav: co je zasazeno, zda je C2 aktivni, zda probehla izolace, zda existuje podezreni na exfiltraci.
3. Management rozhoduje o zakaznickych business vyjimkach, pokud containment zpusobi vypadek kriticke sluzby.
4. Legal / Compliance se zapojuje, pokud je podezreni na unik dat, regulovane udaje nebo povinnost hlaseni.

## 17. PODMINKY UZAVRENI INCIDENTU

Incident lze uzavrit pouze pokud:

- [ ] SOC L2 potvrdil klasifikaci `True Positive / False Positive / Benign / Undetermined`.
- [ ] Ticket obsahuje source host, user, destination, port, protokol, alert ID a timestampy.
- [ ] U P1/P2 je ulozen EDR/XDR process tree, hash, command line a network timeline.
- [ ] U P1/P2 je provedena nebo zdokumentovane odmitnuta izolace hosta.
- [ ] Potvrzene malicious IOC jsou zablokovane nebo je zdokumentovano, proc blokace nebyla provedena.
- [ ] Je overeno, ze C2 komunikace po containmentu nepokracuje.
- [ ] Je overeno, zda doslo nebo nedoslo k exfiltraci dat.
- [ ] U P1/P2 probehl recovery monitoring dle severity.
- [ ] Customer Security potvrdil recovery, pokud byl zasazen zakaznicky system nebo produkcni workload.
- [ ] SOC Service Manager schvalil closure u P1/P2.
- [ ] Closure reason je konkretni a auditovatelny.

**Closure potvrzuje:**
- P4/P3: SOC L2.
- P2: SOC L2 + Customer Security podle dopadu.
- P1: SOC L3 + SOC Service Manager + Customer Security.
